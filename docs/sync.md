# Синхронизация данных

## Принцип работы

Клиент работает офлайн с локальной SQLite базой данных. Сервер — цель
eventual consistency. Синхронизация происходит автоматически при наличии
соединения.

## Определение доступности сервера

Вместо ICMP ping используется выделенный gRPC метод `Ping`, который:
- проходит через тот же стек что и основные запросы
- возвращает серверное время (позволяет детектировать расхождение часов)
- работает через файерволы и прокси

Доступность определяется в каждом цикле синхронизации: если gRPC-вызов
`PushPending` или `Sync` завершается ошибкой — считается что сервер недоступен
и интервал переключается на режим offline.

## Адаптивный интервал синхронизации

| Состояние                    | Интервал |
|------------------------------|----------|
| Есть записи со статусом pending | 10 сек |
| Нет pending записей          | 60 сек   |
| Offline (ошибка соединения)  | 5 мин    |

## Статусы записей

Каждая локальная запись имеет поле `sync_status`:

- **pending** — локальное изменение, ещё не отправлено на сервер
- **synced** — совпадает с версией на сервере
- **conflict** — обнаружен конфликт, требующий внимания

## Алгоритм синхронизации

### Первый логин

```
1. Аутентификация → получаем JWT + kdf_salt
2. Выводим master_key в память
3. GET /secrets — все записи пользователя с сервера
4. Сохраняем в local.db со статусом synced
5. Записываем last_sync_at = now()
```

### Регулярная синхронизация (PUSH → PULL)

**Фаза PUSH — отправка локальных изменений:**

```
1. Выбираем все записи с sync_status = 'pending'
2. Для каждой записи:
   — server_id IS NULL → CreateSecret (новая запись)
   — server_id NOT NULL → UpdateSecret(server_id, payload, server_version)
3. Сервер проверяет: если version != server_version → возвращает CONFLICT
4. Обрабатываем конфликты (см. ниже)
```

**Фаза PULL — получение изменений с сервера:**

```
1. ListSecrets(since: last_sync_at)
2. Сервер возвращает записи изменённые после last_sync_at
3. Для каждой серверной записи:
   — локальный статус synced → upsert серверной версией
   — локальный статус pending → пропускаем (обработано в PUSH)
4. Обновляем last_sync_at = now()
```

## Разрешение конфликтов

Конфликт возникает когда клиент пытается обновить запись, но сервер уже
имеет более новую версию (изменение с другого клиента).

**Стратегия: last-write-wins по `updated_at`**

```
local.updated_at > server.updated_at  → побеждает клиент (force update)
local.updated_at < server.updated_at  → побеждает сервер (локальная перезапись)
равны                                 → побеждает сервер (tiebreak)
```

Разрешение происходит автоматически без участия пользователя. При победе
сервера локальная запись перезаписывается серверной версией.

## Завершение процесса

При получении сигнала завершения (SIGINT / SIGTERM):
1. Остановить фоновые горутины синхронизации
2. Выполнить `wal_checkpoint(TRUNCATE)` — сбросить WAL в основной файл
3. Закрыть соединение с SQLite
4. Очистить master_key из памяти
