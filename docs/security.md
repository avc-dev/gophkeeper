# Модель безопасности GophKeeper

## Принцип zero-knowledge

Сервер никогда не видит данные пользователя в открытом виде. Все секреты
шифруются на клиенте до отправки на сервер. При утечке базы данных сервера
данные пользователей остаются защищены — без мастер-пароля расшифровка
невозможна.

## Иерархия ключей

```
master_password (только в памяти процесса, никогда не сохраняется)
       │
       ├─── bcrypt(password, cost=12) ──────► сервер (проверка при логине)
       │
       └─── Argon2id(password, kdf_salt) ──► master_key (32 байта, AES-256)
                                              только в памяти, не на диске
```

**kdf_salt** — случайные 16 байт, генерируются при регистрации. Хранятся
на сервере и возвращаются клиенту при логине. Нужны для того, чтобы любой
авторизованный клиент мог вывести тот же `master_key` из того же пароля.

### Почему два разных механизма

Аутентификация и шифрование — разные задачи, решаемые по-разному:

- **bcrypt** — проверяет что пользователь знает пароль. Однонаправленная
  функция, из хэша пароль не восстановить.
- **Argon2id** — выводит детерминированный ключ из пароля. Один и тот же
  пароль + соль всегда дают один и тот же ключ на любом устройстве.

Нельзя использовать bcrypt для вывода ключа: он намеренно ограничен 72
байтами и не предназначен для KDF.

## Шифрование данных

Алгоритм: **AES-256-GCM** (Authenticated Encryption with Associated Data).

Даёт одновременно конфиденциальность и целостность — подмена или повреждение
зашифрованных данных обнаруживается при расшифровке.

```
plaintext  = сериализованные данные секрета (JSON)
nonce      = случайные 12 байт (генерируются заново для каждого шифрования)
aad        = []byte(secret.name)
ciphertext = nonce || AES-256-GCM.Seal(master_key, nonce, plaintext, aad)
```

**Additional Authenticated Data (aad)** — имя записи. Защищает от атаки
переноса: нельзя взять зашифрованный пароль с именем "github" и подставить его
под именем "twitter" — GCM-тег не пройдёт проверку. Используется имя, а не ID,
чтобы корректно работать с записями созданными офлайн (до получения server_id).

## Что где хранится

| Данные             | Сервер | SQLite клиента | Память |
|--------------------|--------|----------------|--------|
| bcrypt(password)   | ✓      | ✗              | ✗      |
| kdf_salt           | ✓      | ✓ (копия)      | ✗      |
| master_key         | ✗      | ✗              | ✓ только в сессии |
| JWT токен          | ✗      | ✓              | ✓      |
| Зашифрованный payload | ✓   | ✓              | ✗      |
| Данные в открытом виде | ✗  | ✗              | ✓ только при отображении |

## Защита локального файла базы данных

- Права доступа `0600` (unix) / ACL только для текущего пользователя (windows)
- Расположение в `os.UserConfigDir()` — системный каталог данных пользователя
- SQLite WAL mode с `SetMaxOpenConns(1)` — корректная работа нескольких горутин
- `wal_checkpoint` при завершении процесса — очистка журнала

## JWT подпись

Токены подписываются по алгоритму **EdDSA (Ed25519)**:

- Сервер хранит пару ключей: `private_key` — для подписи, `public_key` — для проверки
- Клиенты никогда не видят приватный ключ
- Ed25519 выбран вместо HMAC-SHA256: асимметричная схема устойчивее к утечке ключа;
  при симметричной схеме любой, кто видел ключ, может выпустить произвольный токен

## Восстановление доступа

**Восстановление мастер-пароля невозможно.**

Это следствие zero-knowledge архитектуры: сервер не хранит мастер-пароль
и не может его восстановить. При утере мастер-пароля все зашифрованные
данные становятся недоступны безвозвратно.

Пользователь предупреждается об этом явным сообщением при регистрации.

Смена мастер-пароля возможна пока пользователь помнит текущий: все записи
перешифровываются новым ключом локально и отправляются на сервер.
