# Улучшения для production

Данный проект реализован как MVP с намеренными упрощениями ради скорости
разработки. Ниже описано что стоит доработать при переходе в production.

## Безопасность

**SQLCipher** — полное шифрование файла SQLite включая метаданные, схему
и WAL. Сейчас payload зашифрован, но имена записей, типы и временные метки
хранятся в открытом виде (metadata leak).

**Шифрование метаданных** — поля `name`, `type`, `metadata` содержат
информацию о структуре данных пользователя. В production их тоже нужно
шифровать.

**mlock()** — запрет свопа страниц памяти содержащих `master_key`. Без
этого ОС может вытеснить ключ на диск в swap-файл.

**Secure memory zeroing** — явная очистка памяти после использования ключа.
Go GC не гарантирует немедленную очистку.

**OS Keychain интеграция** — хранить JWT и `kdf_salt` в macOS Keychain,
Windows Credential Manager или libsecret (Linux) вместо SQLite.

**Account key** (как в 1Password) — второй случайный ключ помимо пароля.
Защищает от brute-force даже при утечке зашифрованной базы.

**Emergency Kit** — PDF с `kdf_salt` и инструкцией, генерируемый при
регистрации. Пользователь распечатывает и хранит физически.

## Архитектура

**DTO на границах слоёв** — сейчас доменные типы передаются между слоями
напрямую. В production изменение в одном слое может неожиданно затронуть
другой. Решение: отдельные DTO для каждой границы с явной конвертацией.

**Интерфейс для Service** — сейчас Service реализован как конкретная
структура. Это затрудняет изолированное тестирование Handler. Решение:
добавить интерфейс для Service, мокировать в тестах Handler.

**CQRS** — разделить операции чтения и записи на уровне Service. Сейчас
один Service смешивает read и write операции, что усложняет масштабирование.

**Типизированные ошибки** — сейчас используется `fmt.Errorf`. В production
нужны machine-readable коды ошибок для корректной обработки на клиенте
и структурированного логирования.

**Observability** — structured logging (zap/zerolog), метрики (Prometheus),
распределённая трассировка (OpenTelemetry).

**Audit log** — отдельный лог кто, когда и какие данные запрашивал.
Отделить от application log.

## Функциональность

**Binary streaming** — сейчас бинарные данные ограничены 4MB (лимит gRPC
сообщения). Для больших файлов нужен streaming upload/download.

**Key rotation** — смена мастер-пароля должна перешифровывать все записи.
Сейчас не реализовано.

**История изменений** — версионирование записей с возможностью откатиться
к предыдущей версии.

**Sharing** — передача отдельных записей другому пользователю с шифрованием
на его публичный ключ.

**Import** — импорт данных из популярных менеджеров паролей (1Password,
Bitwarden, LastPass, KeePass).

**Real-time sync** — gRPC server-side streaming как замена polling. Сервер
пушит изменения клиенту в реальном времени вместо опроса раз в N секунд.

**Windows installer** — `.msi` пакет с правильными ACL на файл базы данных
при установке.
